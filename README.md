### 📌 Gitflow 브랜치 전략

우리 팀은 **Gitflow** 워크플로우를 도입하여 효과적인 협업과 안정적인 코드 관리를 지향합니다. 아래의 브랜치 규칙을 준수하여 개발을 진행해주세요.

* **`main`**: 프로덕션 배포가 가능한 안정적인 코드를 관리합니다. 이 프로젝트는 해당 부분을 release와 통합하여 사용합니다.
* **`develop`**: 다음 배포를 위한 개발이 이루어지는 주 브랜치입니다.
* **`feature/*`**: 새로운 기능 개발을 위한 브랜치입니다. `develop`에서 분기하여 작업 완료 후 `develop`으로 병합합니다. (예: `feature/user-login`, `feature/diary-crud`)
* **`fix/*`**: 버그와 같은 오류 수정을 위한 브랜치입니다. `develop`에서 분기하여 오류 수정작업을 진행합니다.
* **`hotfix/*`**: `main`에서 발생한 긴급 버그를 수정하는 브랜치입니다. `main`에서 분기하며, 수정 완료 후 `main`과 `develop`에 모두 병합합니다.

### 📚 커밋 메시지 컨벤션

아래의 커밋 메시지 컨벤션을 준수하여 커밋 이력을 명확하게 관리합니다.

| 메시지         | 설명 |
|:------------| :--- |
| **`feat`**  | 🥥 새로운 기능 추가 또는 기존 기능 수정 |
| **`fix`**   | 🐛 버그 수정 |
| **`docs`**  | 📝 문서(주석, README 등) 수정 |
| **`ref`**   | ✏️ 기능 변화가 없는 코드 리팩토링 |
| **`test`**  | ⚙️ 테스트 코드 추가 또는 수정 |
| **`chore`** | 🛠️ 빌드 시스템, 라이브러리, 환경설정 등 기타 수정 |

**💡 커밋 메시지 예시**
* `feat: 로그인 폼 완성`
* `fix: 회원가입 에러 해결`
* `docs: 회원가입 로직 주석 수정`
